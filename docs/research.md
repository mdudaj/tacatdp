# TACATDP Power Apps Research Notes

## Goal

Convert the TACATDP XLSForm into a Microsoft Power Apps canvas application that preserves ODK-like behavior, including skip patterns, required fields, constraints, choice lists, grouped sections, and user-friendly validation, then extend the experience for Microsoft Lists-backed data collection.

The workbook includes seven ODK runtime fields to exclude from the Power Apps data-entry flow: `starttime`, `endtime`, `deviceid`, `subscriberid`, `simid`, `devicephonenum`, and `username`.

Notes, calculations, groups, and ODK runtime metadata are excluded from the maintained field inventory because they are not data-entry fields. Notes should be treated as labels, section headings, or helper text in the Power Apps UI.

These should not be implemented as static in-app choice literals. They should be backed by Microsoft Lists lookup/reference data or loaded into collections from separate reference lists.

## Platform findings

### Canvas app source control and packaging

Microsoft canvas apps can be exported as `.msapp` packages and represented as source-friendly YAML files (`*.pa.yaml`). The current repository already contains this YAML-style source. Microsoft documentation warns that canvas YAML source is intended mainly for review and minor edits; larger app work should be validated by packing/importing through supported Power Platform tooling.

The recommended implementation loop is:

1. Keep exported source under version control.
2. Edit source carefully in small, reviewable changes.
3. Repack to `.msapp` with Power Platform CLI tooling.
4. Import or open the packed app in Power Apps Studio.
5. Validate behavior in Studio against the Microsoft Lists data source.

Useful tooling path:

```bash
pac canvas pack --sources app-src --msapp build/TACATDP.msapp
pac canvas unpack --msapp build/TACATDP.msapp --sources app-src
```

The exact commands may need adjustment after confirming whether the current `.msapr` export is a direct canvas source folder, a Power Apps Studio project artifact, or part of a solution-oriented workflow.

### Microsoft Lists integration

Power Apps can use Microsoft Lists through the SharePoint connector. Generated forms such as `frmSubmission` can use `SubmitForm`-style behavior, which automatically respects data card validation for visible required fields. If the app uses `Patch` for custom multi-screen or custom validation flows, required checks and data type checks must be implemented explicitly before saving.

For this project, the safer default is:

- Use form data cards where possible for list-bound fields.
- Use explicit validation formulas for XLSForm constraints and cross-field logic.
- Gate navigation and submission with a centralized validation state.
- Surface errors close to fields and with summary notifications.

### XLSForm behavior to preserve

The XLSForm workbook should be treated as the source of truth for survey behavior:

- `survey` rows define fields, groups, labels, hints, required status, calculations, constraints, and relevance rules.
- `choices` rows define options for `select_one` and `select_multiple`.
- `relevant` expressions define skip patterns.
- `constraint` and `constraint_message` define validation rules.
- `required` defines mandatory fields.
- ODK metadata fields such as `start`, `end`, `deviceid`, `subscriberid`, `simserial`, `phonenumber`, `username`, and similar collector/runtime-only fields should be excluded from the Power Apps data-entry experience unless explicitly retained for audit purposes.

## Key design implications

1. ODK-only metadata fields should be removed from the app and Microsoft Lists schema, then replaced with Microsoft 365 audit fields where needed.
2. Input choice lists need an explicit mapping to Power Apps controls and Microsoft Lists column types.
3. Large choice lists and cascading choice filters should be normalized into reference lists rather than embedded into formulas.
4. Skip logic should drive each data card or section `Visible` formula.
5. Constraints should be implemented as Power Fx formulas and tested against known valid/invalid cases.
6. Long forms should not remain as one giant generated screen; group structure from XLSForm should become sections or screens.
7. Microsoft Lists column limits, choice columns, lookup fields, and attachment needs must be confirmed before final schema generation.
